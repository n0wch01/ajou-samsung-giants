# Guardrail vs Direct (SSOT)

OpenClaw를 **같은 시나리오**로 돌릴 때, 설정 프리셋에 따라 관측되는 행동이 달라진다. SG 문서·런북·QA는 아래 두 라벨 중 하나를 **반드시** 밝힌다.

## Guardrail(권장·운영 지향)

**의도:** 에이전트가 할 수 있는 일을 최소화하고, 위험한 툴·플러그인 경로는 기본적으로 막거나 **승인** 후에만 허용한다.

| 영역 | 일반적인 설정 방향 |
|------|---------------------|
| 샌드박스 | 허용된 작업 디렉터리·네트워크만 사용 |
| 툴 정책 | allow/deny 목록, 플러그인 출처별 제한 |
| 승인 | 민감 `session.tool` 호출 전 사용자 확인 |
| 비밀 | 환경 변수·파일 최소 노출; `config.get` 스냅샷으로 문서화 |

**문서화:** 런북 2단계(가드레일 검사)에 기대 결과를 적는다. 예: 미승인 플러그인 툴은 `tools.effective`에 없거나, 호출 시 거부/대기.

## Direct(교육·랩 전용)

**의도:** 가드레일을 **의도적으로 완화**해, “막지 않았을 때 무엇이 보이는가”를 교육한다.

| 항목 | 요구사항 |
|------|-----------|
| 운영 | **프로덕션·공용 워크스테이션에서 사용 금지** |
| 동의 | 팀이 지정한 격리 호스트·계정에서만 수행 |
| 기록 | 동일 S1이라도 Guardrail 실행과 **대비**해 로그·스냅샷을 남긴다 |

**문서화:** Direct 실행에는 런북에 “운영 금지” 경고 문구를 넣고, `tools.effective`에 플러그인 툴이 노출될 수 있음을 기대값으로 명시한다.

## 스냅샷·증거

- **Guardrail:** `config.get`(또는 팀 표준 스냅샷)으로 정책 요약을 붙인다.
- **둘 다:** `tools.catalog` / `tools.effective` 전후 덤프, `session.tool` JSONL(Sentinel `trace.jsonl`)을 증거로 남긴다.
